Wrapper starter

A suggested starting point for creating cross-platform artisinal charts with default FT styling

How is this structured?

Visual Vocabulary templates are generally divided between four separate files:

  • index.html — This instantiates the chart and contains markup for each of the chart frames. If you use wrapper-starter as a starting point, you don't need to change anything, though it's recommended you update the documentation (i.e., the section you're reading right now) to explain how to use your template.
  • chartNameHere.js — You should rename this to whatever your chart name is ("choropleth.js"; "line.js"). All of your chart drawing code lives in here. It's effectively a factory function (I.e., a function that returns another function) that exposes a bunch of methods for manipulating the internal properties of your chart that might not necessarily be user-facing (By "user" here we mean "person creating charts using the template," not the reader) through "getters" and "setters". In order to chain these calls together, we use the fluent API style of programming. The general way to write one of these is:
    
                chart.yDomain = (d) {
                    if (d === undefined) return yScale.domain(); // The "getter" part. Return the current domain if no argument.
                    yScale.domain(d); // The "setter" part. If argument present, set the associated value to it.
                    return chart; // The fluent API part. Always return the parent chart function itself.
                }

    Note you can totally delete many of the existing getters/setters in wrapper-starter — if you don't use it or it's not useful, kill it!
    The main part is the actual chart() function prototype, which is where you pass data to a D3 selection and append the relevant elements to the parent group, passed as the parent argument in the function. Look at the other templates for a good example how to do this.
  • index.js — This is where all your user-facing code lives. Any variables (filenames for data, initial values, per-frame customisations, et cetera) should be in here.
    Make sure to:
    • Change ln. 3 to reflect the name of your chart as per the above, both in terms of filename and what you want to expose it as in the code
    • Update the default constants starting around ln. 15 to reflect your chart type
    • Update ln. 80 to reflect the namespace you used on ln. 3
  • parseData.js — This is where all the heavy data munging happens. You probably want to edit the main body of fromCSV to handle this.
